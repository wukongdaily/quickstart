# AI Development Guide

## 1. 文档目标与适用范围

这份文档用于帮助 AI 在 QuickStart 项目中开展日常开发、排查问题和重构准备工作。它不是面向终端用户的产品说明，也不是完整架构文档，而是一份面向开发过程的工作指引。重点是帮助 AI 快速建立正确上下文，减少前后端边界误判，避免在设备相关代码上做脱离环境假设的修改。

文档覆盖整个 QuickStart 项目，而不是单一目录。当前仓库内至少有两个需要联动理解的核心目录：

- 后端目录 `backend`，Swagger / YAML 接口源文件在 `backend/yamls`
- 前端目录 `web`

这份文档主要服务以下场景：

- 定位某个功能应该在哪个目录或层修改
- 判断一个变更是否会同时影响前端、后端和 LuCI 挂载方式
- 在任务测试或远程部署失败时，快速定位是前端、LuCI 代理层、后端二进制，还是目标设备环境的问题
- 在修改 API、页面流程或系统行为前，先识别依赖链和副作用范围
- 为后续大规模重构提供稳定的模块边界和风险提示

这份文档不替代源码阅读。它的作用是提供优先级、入口和约束，帮助 AI 先走对方向，再进入具体代码。

## 2. AI 快速工作规则

在 QuickStart 项目中工作时，AI 应先判断当前任务落在哪一层，再进入具体实现。这个项目不是单目录单层结构，很多功能变更会同时涉及前端页面、后端 API、LuCI 页面入口，以及设备侧系统行为。先判断边界，比直接改代码更重要。

处理任务时，建议遵循以下顺序：

1. 先判断任务属于哪一类
   是前端展示问题、后端接口问题、前后端联动问题，还是设备系统行为问题。不要在没有分类的情况下直接搜索和修改。
2. 先确认修改目录
   页面、路由、组件、前端请求逻辑通常在 `web`。
   API 路由、业务逻辑、系统命令、`ubus/uci` 交互通常在 `backend`。
3. 涉及接口时，始终做双向核对
   前端要看请求封装、调用路径、数据结构。
   后端要看路由注册、service 入口、模型定义，以及 `backend/yamls` 下的 Swagger/YAML 源定义。
   不要只改一侧就假设另一侧会自动兼容。
4. 涉及系统行为时，默认假设有副作用
   网络、磁盘、RAID、SMART、共享、无线、Docker 相关逻辑都可能直接影响设备状态。
   修改前先定位它最终调用的是普通内部函数，还是 `ubus`、`uci`、shell 命令。
5. 不要把设备相关功能当作普通本地 Web 项目来验证
   这个项目很多逻辑依赖真实 OpenWrt / iStoreOS 环境。
   本地能通过编译，不等于真实设备行为正确。
   如果无法在当前环境验证，应明确指出验证边界。
6. 修改前先评估前后端联动影响
   以下情况通常需要同时检查 `web` 和 `backend`：
   - 新增或修改 API
   - 页面流程变更
   - 配置字段变更
   - 首次启动、向导、挂载、共享、设备状态展示相关逻辑变更
7. 涉及任务测试或远程调试时，优先使用仓库内已有 `.it-runner` 任务
   这个项目已经包含面向目标设备的 `.it-runner` 配置和任务集合。
   在需要构建、部署、重启、看状态、看日志、做诊断时，优先先看 `.it-runner/README.md`、`.it-runner/project.yaml` 和 `.it-runner/tasks/*`，不要重复发明一套临时命令。
8. 处理请求链路问题时，不要忽略 LuCI 代理层
   浏览器访问的并不是直接暴露在 80 端口上的 Go 服务。
   真实链路里还包括 `uhttpd` 和 LuCI controller 反向代理。
   如果页面请求异常、接口 404、认证失败或代理报错，应把 LuCI controller 文件纳入排查范围。
9. 重构时优先保留外部行为稳定
   在没有明确目标时，不要同时改变接口路径、返回结构、页面流程和系统命令行为。
   先拆边界，再改行为，比直接大范围重写更安全。
10. 遇到生成内容时先确认来源
   如果模型、接口类型或文档来自 Swagger/YAML 生成，不要直接把生成产物当成纯手写代码处理。
   先确认源文件，再决定改动入口。

## 3. 任务分流：不同类型任务应先看哪里

面对 QuickStart 项目的开发任务时，AI 应先做任务分流，再进入具体代码。这个项目跨前端目录、后端目录和设备运行环境，同一种表面问题，真实修改入口可能完全不同。先判断任务类型，可以显著减少误改。

### 3.1 页面展示或交互问题

如果问题表现为页面布局、组件交互、路由跳转、弹窗、表单、状态展示异常，优先从前端目录 `web` 开始排查。重点查看：

- `web/src/pages/*` 下对应页面
- `web/src/components/*` 下复用组件
- `web/src/plugins/router/index.ts` 路由入口
- `web/src/request/*` 请求封装
- `web/src/types/*` 类型定义

这类问题只有在前端请求数据结构与后端返回不一致时，才需要继续追到后端。

### 3.2 后端接口或业务逻辑问题

如果问题表现为接口报错、返回值异常、接口缺失、业务流程不符合预期，优先从后端目录 `backend` 开始排查。重点查看：

- `backend/server/*.go` 中的路由注册
- `backend/service/*.go` 中的业务逻辑入口
- `backend/models/*` 中的响应和请求模型
- `backend/yamls/*.yaml` 中的接口定义

这类任务通常需要确认前端是否依赖了某个固定字段或固定错误处理方式。

### 3.3 前后端联动问题

如果问题涉及“前端调用成功但页面不对”“页面参数发出去了但设备状态没变”“后端改了接口但前端没同步”，应同时查看两侧代码。排查顺序建议是：

1. 前端请求路径与参数
2. 后端路由是否匹配
3. service 是否正确处理
4. 返回结构是否与前端消费方式一致

不要只根据接口名猜测联动关系，要用实际调用链确认。

### 3.4 设备系统行为问题

如果任务涉及网络配置、磁盘初始化、分区格式化、RAID、SMART、共享、无线、Docker、DDNS、首次向导等，默认视为设备系统行为问题。这类任务除了看 API 和页面外，还要继续下钻到：

- `ubus` 调用
- `uci` 配置读写
- shell 命令拼接与执行
- 设备特定脚本或平台分支逻辑

这类代码修改风险高，可能影响设备连通性、存储状态或服务可用性。不能只以“接口返回正常”作为验证标准。

### 3.5 文档、模型或接口定义问题

如果任务涉及类型漂移、接口文档不一致、模型字段来源不清，优先确认它是手写结构还是生成产物。当前项目中，部分模型与接口定义来自 YAML / Swagger 流程。遇到这类问题时，应优先查源定义，再决定是否修改生成结果。

### 3.6 重构准备类任务

如果任务目标不是修一个点，而是“梳理模块”“准备拆分”“抽边界”“降低耦合”，不要一开始就改实现。应先做三件事：

- 画出当前调用链
- 标出前后端联动依赖
- 标出直接操作设备状态的副作用点

重构类任务的首要目标不是减少代码行数，而是先建立稳定边界。

## 4. 项目全景：目录分工、运行链路与 LuCI 挂载关系

QuickStart 不是一个单目录、单进程、单前端入口的普通 Web 项目。它由多个层次拼接而成，开发时需要同时理解“代码在哪个目录或层”“页面如何挂到设备后台”“请求最终如何落到设备系统行为”这三件事。

### 4.1 目录分工

当前仓库内至少有两个核心目录需要联合理解：

- `backend`
  后端目录。主要负责：
  - HTTP / Unix Socket 服务入口
  - LuCI 侧调用的 API 路由
  - 业务 service 层
  - `ubus`、`uci`、shell 命令封装
  - `backend/yamls` 中的 Swagger / YAML 接口定义和部分模型来源
- `web`
  前端目录。主要负责：
  - Vue 页面和组件
  - 路由与页面流程
  - 前端请求封装
  - 页面状态管理
  - 打包产物输出到 LuCI 静态资源目录
- `istoreos-app-hub/apps/quickstart/luci-app-quickstart`
  LuCI 集成仓库。至少包含把浏览器请求代理到后端二进制的 controller 文件。
  其中 `luasrc/controller/istore_backend.lua` 是排查请求链路问题时的重要入口。

AI 在分析问题时，不应只根据“这是页面问题”或“这是接口问题”做单侧假设。很多问题的真正边界在 `web`、`backend` 和 LuCI 集成层之间。

### 4.2 页面运行链路

QuickStart 前端不是作为独立站点运行在浏览器中，而是挂载到 LuCI 页面体系中。典型开发链路是：

1. LuCI 模板页面负责输出 HTML 壳
2. 模板中注入运行时变量，例如：
   - `window.token`
   - `window.vue_base`
   - `window.vue_lang_data`
   - `window.vue_lang`
3. 页面再加载 `web` 开发服务器或构建产物中的前端入口
4. 前端根据 `vue_base` 和当前路由渲染对应页面
5. 浏览器将 `/cgi-bin/luci/istore/...` 请求交给设备上的 `uhttpd`
6. `uhttpd` 将请求交给 LuCI controller 处理
7. `luci.controller.istore_backend` 再把请求代理到本机 `127.0.0.1:3038`
8. quickstart 后端二进制接收请求并继续调用业务逻辑
9. 业务逻辑再下钻到设备系统层

因此，前端并不是“独立控制一切”的应用层，它依赖 LuCI 注入的上下文，也依赖设备侧路径约定。

### 4.3 LuCI 代理层的实际位置与作用

真实设备上，请求链路中的 LuCI 代理层是一个需要单独理解的组成部分。当前已知的重要文件包括：

- 开发侧路径：
  `/projects/workspace-linkease-ubuntu/openwrt-apps/istoreos-app-hub/apps/quickstart/luci-app-quickstart/luasrc/controller/istore_backend.lua`
- 目标系统路径：
  `/usr/lib/lua/luci/controller/istore_backend.lua`

这个 controller 做的不是简单路由注册，而是带会话透传的反向代理。当前文件中的关键行为包括：

- 通过 `entry({"istore"}, call("istore_backend"))` 注册 LuCI 入口
- 监听并接收来自 `uhttpd` 的请求
- 从 `sysauth_*` cookie 中恢复 LuCI session
- 通过 `ubus session get` 校验 session
- 将 `X-Forwarded-Sid`、`X-Forwarded-Token` 等头转发给后端
- 把请求代理到 `127.0.0.1:3038`
- 再把响应头和响应体返回给浏览器

这意味着只看前端和 Go 后端还不够。以下问题都可能出在 LuCI 代理层：

- 浏览器能打开页面，但接口全部 403
- 接口路径看起来正确，但实际返回 404 或 500
- 登录态在页面里失效，但后端服务本身仍然存活
- 本机直接访问 `127.0.0.1:3038` 正常，但浏览器经 LuCI 访问异常

涉及这些现象时，应把 `istore_backend.lua` 作为一等排查对象。

### 4.4 开发模式与发布模式的区别

开发模式下，LuCI 模板通常直接引用本地 Vite 服务资源，用于前端热更新和调试。
发布模式下，前端资源会被构建并打包成 LuCI 静态文件，供设备正式页面使用。

这意味着：

- 开发环境中的资源路径与正式发布路径可能不同
- 本地前端可运行，不代表设备页面集成无误
- 修改前端资源路径、构建输出位置或 LuCI 模板变量时，应视为高风险改动

### 4.5 `.it-runner` 是任务测试和远程运维链路的一部分

当前仓库已经包含一套可提交的 `.it-runner` 配置，它不是临时目录，而是项目约定的一部分。它的主要用途不是本地 `go run`，而是围绕“构建本地二进制 -> 部署到目标设备 -> 控制远端 quickstart 服务 -> 收集状态和日志”这一条远程设备工作流。

当前可直接利用的信息包括：

- `.it-runner/project.yaml`
  定义任务目录、日志目录和缓存目录
- `.it-runner/envs/000-defaults.env`
  存放单目标设备的非机密默认连接信息和远端路径
- `.it-runner/tasks/*`
  存放构建、部署、重启、看状态、看日志、验证、诊断、发布、测试等任务
- `.it-runner/README.md`
  说明这套任务的设计目的，以及哪些路径属于本地运行产物

AI 在任务测试和故障定位时，应优先把 `.it-runner` 看成项目内置的标准操作面，而不是附属资料。尤其当任务涉及以下场景时：

- 构建目标设备可运行的 quickstart 二进制
- 上传并安装新二进制
- 重启目标设备上的 quickstart 服务
- 检查远端服务状态、进程、二进制路径
- 收集 `logread` 日志进行诊断

如果遇到“本地代码看起来正常，但设备行为不对”的情况，`.it-runner` 相关任务往往是最快的第一手排查入口。

### 4.6 请求链路与设备行为链路

从请求视角看，QuickStart 的完整链路通常是：

`LuCI 页面 -> Vue 前端 -> uhttpd:80 -> luci.controller.istore_backend -> quickstart:127.0.0.1:3038 -> Go server -> service -> ubus/uci/shell -> 设备状态变化`

这条链路决定了几个事实：

- 前端问题不一定是前端问题，后端返回结构变化也会导致页面异常
- LuCI controller 代理异常也会表现成“像是后端挂了”或“像是登录态失效”
- 后端接口正常返回不等于设备已经正确生效
- 某些变更会跨越“页面逻辑”和“系统副作用”两个层次，验证时不能只看其中一侧

### 4.7 AI 在项目全景中的基本判断原则

AI 处理任务时，建议先回答以下问题：

- 当前问题发生在页面层、接口层，还是设备行为层？
- 当前需求落在哪个目录或层，是否存在 LuCI 集成仓库或另一侧业务代码的配套改动？
- 这次改动是否会影响 LuCI 挂载方式、前端打包产物或 API 路径约定？
- 这次改动是否会影响 `.it-runner` 任务约定、目标设备部署方式或远端诊断路径？
- 这次改动最终是否会触发真实设备配置变化？

只有先回答这些问题，后面的实现和重构判断才可靠。

## 5. 模块地图：功能域、主要入口与典型风险

QuickStart 后端和前端并不是按完全一致的文件结构组织，但整体功能域比较稳定。AI 在处理需求时，建议先按功能域定位，再进入具体文件。下面这份模块地图不是完整清单，而是用于快速判断“该先看哪里”和“这类改动常见风险是什么”。

### 5.1 `network`

主要职责：

- 网络状态展示
- 接口状态与端口信息
- 设备列表
- 公网连通性检查
- Homebox、接口配置等网络能力

前端通常关注：

- 控制台首页中的网络信息组件
- 网络设置向导页面
- 接口配置与测速相关页面

后端通常关注：

- `server/routers.go` 中 `network` 路由
- `service/network*.go`
- 网络接口与设备状态相关模型
- 可能涉及 `ubus`、`uci network`、`firewall`

典型风险：

- 修改网络配置可能直接影响设备连通性
- 返回字段变化容易同时影响首页卡片、向导页面和接口配置页
- 仅凭前端展示异常不能判断是页面问题还是接口结构变化

### 5.2 `system`

主要职责：

- 系统版本
- 时间、CPU、内存、总体状态
- 升级检查
- 自动检查更新开关
- 重启、关机、密码等系统操作
- 模块设置

前端通常关注：

- 控制台首页状态卡片
- 系统更新提示
- 模块设置弹窗或配置区域

后端通常关注：

- `service/system.go`
- 版本号与设备状态组装逻辑
- 某些平台相关温度、状态读取逻辑

典型风险：

- 某些字段依赖真实设备环境，不能在普通开发机完整复现
- 重启、关机、密码相关变更属于高副作用操作
- 状态页可能聚合多个后端来源，不宜只盯单个函数

### 5.3 `nas`

主要职责：

- 磁盘状态
- 初始化、挂载点、分区格式化与挂载
- WebDAV、Samba、LinkEase
- 可能包括沙箱或存储初始化相关流程

前端通常关注：

- 首页存储卡片
- 存储服务配置组件
- 动作弹窗，例如格式化、挂载、创建共享

后端通常关注：

- `service/nas.go`
- `service/share.go`
- `service/primitive.go`
- 分区、挂载、共享协议、磁盘状态相关模型

典型风险：

- 直接作用于磁盘和挂载状态
- 很多动作不可逆或会影响已有数据
- 返回结果除了接口成功，还要关注系统是否真正生效

### 5.4 `guide`

主要职责：

- 首次安装或首次进入时的向导流程
- PPPoE、DHCP、旁路由/网关模式
- 下载器初始化
- Docker 数据迁移或切换
- DDNS、软件源、全局目录等初始化能力

前端通常关注：

- 引导页面
- 分步表单
- 一次性动作弹窗和状态反馈

后端通常关注：

- `service/guide.go`
- 与网络、存储、Docker、DDNS 相关的桥接逻辑
- `quickstart` 配置中记录的安装/初始化状态

典型风险：

- 这是跨模块最重的区域之一，经常同时联动网络、磁盘、服务和配置状态
- 很多逻辑是“流程编排”，不是单一资源读写
- 重构时最容易出现页面流程正确但设备状态未完成的假成功

### 5.5 `raid`

主要职责：

- RAID 创建、删除、增删成员、恢复、详情、自动修复、可创建列表

前端通常关注：

- RAID 页面
- 创建与恢复相关弹窗
- RAID 状态展示组件

后端通常关注：

- `service/raid.go`
- RAID 请求/响应模型
- 可能关联块设备、阵列状态、系统工具调用

典型风险：

- 属于高风险设备操作
- 不能把“接口返回成功”当成阵列状态稳定的充分条件
- 修改时应明确区分“开始执行”“执行中”“执行完成”的语义

### 5.6 `smart`

主要职责：

- SMART 磁盘列表
- 日志、配置、测试任务、测试结果、属性结果、扩展结果

前端通常关注：

- SMART 页面、日志页、任务页
- 磁盘详情与检测任务组件

后端通常关注：

- `service/smart.go`
- SMART 配置与任务处理逻辑
- 磁盘检测结果模型

典型风险：

- 结果往往具有异步性或延迟性
- 页面展示常常依赖多个结果接口组合
- 测试任务和配置修改不应与普通状态查询混为一谈

### 5.7 `share`

主要职责：

- 共享用户管理
- 共享服务管理
- Samba / WebDAV 协议配置

前端通常关注：

- 用户列表
- 共享服务列表
- 协议配置表单

后端通常关注：

- `service/share.go`
- 用户、服务、权限、协议配置相关模型

典型风险：

- 这类改动通常带有配置持久化与服务重载副作用
- 很容易出现“表单结构改了，但后端兼容逻辑没改”的问题
- 用户、服务、协议三个层次应分开理解

### 5.8 `wireless`

主要职责：

- 无线接口列表
- 启停接口
- 发射功率设置
- 无线编辑与快速配置
- 平台差异化无线处理逻辑

前端通常关注：

- 快速 Wi-Fi 页面
- 无线配置表单
- 快捷设置动作

后端通常关注：

- `service/wifi.go`
- 平台分支文件，例如 QCA 或特定机型实现
- `uci wireless`、无线扫描和设备能力判断

典型风险：

- 无线逻辑常常有明显的平台分支
- 同一套页面未必适用于所有设备实现
- 重构时不要轻易把平台特化逻辑“统一抽象”，除非先验证行为一致性

### 5.9 `quickstart` 配置自身

主要职责：

- QuickStart 自身配置读写
- 首次设置状态、模块开关、引导状态等项目内部配置

后端通常关注：

- `quickstart/get/set/delete` 相关接口
- `service` 中读取 `quickstart` 配置的逻辑
- 可能依赖 `uci` 中的 `quickstart` 配置段

典型风险：

- 虽然名字看起来像普通配置，但它往往控制引导流程和模块行为
- 修改字段时要同步确认前端依赖与默认值处理
- 不应把它当成与业务无关的附属配置

## 6. 跨前后端联动规则：哪些变更不能只改一边

QuickStart 的很多功能并不收敛在单一目录内。前端目录、后端目录、LuCI 页面挂载方式以及设备运行环境之间存在稳定但隐性的耦合。AI 在修改时，不应默认“这只是页面问题”或“这只是接口问题”，而应先判断这次变更是否属于前后端或跨层联动场景。

### 6.1 修改 API 时，默认检查前后端两侧

只要涉及新增接口、删除接口、修改路径、修改请求参数、修改响应结构，建议同时检查：

- 前端请求封装是否已有调用
- 页面是否依赖某个字段名、布尔值、错误码或默认值
- 后端路由是否与前端请求路径一致
- `backend/yamls` 中的 Swagger / YAML 是否需要同步更新
- `models` 是否来自生成流程

不要只在后端改出“更合理”的字段结构，而不确认前端消费方式。

### 6.2 修改页面流程时，默认检查后端行为语义

页面上的一步操作，后端未必只是一次普通数据提交。尤其是向导、网络、存储、共享、RAID、SMART 等页面，很多按钮背后对应的是一次设备动作或多步编排流程。
因此页面流程变更时，建议确认：

- 当前页面动作是否依赖固定接口顺序
- 是否存在“发起动作”“轮询状态”“显示完成”的多阶段语义
- 前端是否把某些后端错误信息翻译成了固定 UI 行为

不要只看组件交互是否顺畅，还要确认设备侧流程语义有没有被破坏。

### 6.3 修改配置字段时，默认检查读写两侧

QuickStart 中有不少配置字段既在前端表单中出现，也在后端 service 或 `uci` 配置里落地。
字段变更时，建议同时检查：

- 前端类型定义
- 请求体拼装方式
- 后端模型定义
- service 中的字段读取与默认值处理
- 持久化配置位置

字段名看似很小的调整，往往会导致“页面提交成功但实际未生效”的隐性问题。

### 6.4 修改 LuCI 集成方式时，默认检查前端启动条件

前端并不是独立站点，它依赖 LuCI 页面注入运行时变量和资源入口。
只要涉及以下内容，建议按高风险跨层改动处理：

- `main_dev.htm` 或正式 LuCI 模板
- `window.vue_base`
- 语言包路径
- 前端构建输出路径
- 开发代理与正式资源路径切换
- `istore_backend.lua` 中的代理逻辑与透传头

这类改动同时影响开发体验、页面路由、资源加载和发布集成，不能只在 `web` 局部验证。

### 6.4.1 修改请求链路时，默认把 `uhttpd -> LuCI controller -> quickstart` 看成整体

如果任务涉及接口可达性、鉴权、路径重写、反向代理、端口监听、请求头透传或响应体转发，不要只检查某一个节点。建议至少同步确认：

- `uhttpd` 暴露的 LuCI 入口是否仍然存在
- `istore_backend.lua` 是否仍将请求代理到正确的目标地址和端口
- LuCI session 是否仍能正确转换为 `X-Forwarded-Sid` / `X-Forwarded-Token`
- quickstart 二进制是否仍在预期端口上监听

这类问题在用户视角里通常表现为“页面接口都坏了”，但真实故障点可能完全不同。

### 6.5 修改设备动作时，默认检查前端反馈是否仍然成立

如果后端把某个动作从同步改成异步，或把返回结构从“成功即完成”改成“成功即开始执行”，前端展示逻辑往往也需要一起调整。
这类场景在 RAID、SMART、磁盘初始化、服务启停、Docker 迁移等功能中特别常见。
因此，设备动作语义一旦变化，建议同步检查：

- 前端提示文案
- 是否需要轮询或等待状态
- 是否还适合沿用原有成功/失败判定方式

### 6.6 重构时优先标记联动边界，而不是一次性抹平差异

跨前后端重构最常见的风险，是试图一次性把接口、页面和设备行为都改成“更统一”的形式。
更稳妥的做法是：

- 先明确哪些边界是前端边界
- 哪些边界是后端 API 边界
- 哪些边界是设备适配边界
- 哪些字段和路径已经被另一侧代码或发布流程隐式依赖

在没有把这些联动边界标清之前，不建议做大范围统一抽象。

## 7. 高风险区域与副作用提醒

QuickStart 不是纯展示型管理台。它的后端包含大量会直接改变设备状态的逻辑，某些操作还会影响网络可达性、磁盘数据、服务可用性或首次安装流程。AI 在这些区域工作时，不应把修改理解为普通业务代码调整，而应先识别副作用类型，再决定修改方式和验证范围。

### 7.1 网络配置相关改动属于高风险操作

涉及以下内容时，应默认高风险：

- LAN / WAN 配置
- PPPoE、DHCP、旁路由/网关模式
- 接口桥接、端口分配、防火墙区关联
- `network`、`firewall`、`dhcp` 相关 `uci` 配置
- 影响设备默认访问地址的行为

原因是这类修改可能直接导致：

- 设备失联
- 页面访问路径变化
- 上游网络不通
- 前端还能发起请求，但后续设备状态已不可恢复地偏离预期

因此，网络改动不能只看代码逻辑是否更整洁，也不能只看接口是否返回成功。

### 7.2 磁盘、分区、RAID 相关改动属于高风险操作

涉及以下内容时，应默认高风险：

- 磁盘初始化
- 分区格式化
- 挂载点调整
- RAID 创建、删除、恢复、自动修复
- 沙箱环境、数据迁移、Docker 存储切换

这类行为可能直接影响数据安全、挂载状态和后续服务依赖。
AI 不应把这类逻辑当成普通表单提交处理，也不应仅根据接口命名推断动作是否可逆。
如果看不清副作用，应继续追到命令执行层再判断。

### 7.3 共享、无线、服务启停类改动属于中高风险操作

以下区域虽然不一定直接破坏设备可达性，但仍有明显副作用：

- Samba / WebDAV 配置
- 共享用户与权限
- 无线接口启停、SSID、加密、发射功率
- 下载器初始化
- Docker、LinkEase、Homebox 等服务启用或切换

这类改动常见问题不是“编译失败”，而是：

- 配置写入了但服务没有重载
- 页面显示成功但服务状态未同步
- 某些设备平台可用，另一些平台不可用
- 前端沿用了错误的成功判断

### 7.4 引导流程和项目自身配置属于隐性高风险区域

首次启动、向导流程、`quickstart` 自身配置看起来不像系统底层，但它们经常决定：

- 页面第一次进入哪个流程
- 哪些模块是否显示
- 某些初始化动作是否重复执行
- 某些系统切换是否被视为已完成

因此，这类配置字段虽然可能很小，但一旦处理不一致，就会造成“页面流程错乱”或“设备状态与 UI 认知不一致”的问题。
不要因为它们名字普通，就低估其影响范围。

### 7.5 平台差异逻辑需要单独看待

项目中存在明显的设备平台分支，尤其体现在：

- CPU 温度或系统状态读取
- 无线能力与无线配置
- 特定机型相关脚本
- 某些 shell 命令或路径依赖

这意味着：

- 在一台设备上成立的行为，不一定可泛化到所有平台
- 某些代码看起来重复，实际是在兼容不同硬件或固件差异
- 重构时不应先以“消除重复”为目标，而应先确认行为是否真的一致

### 7.6 无法完整验证时，应明确说明验证边界

对于真实依赖设备环境的改动，AI 如果只能做到静态阅读、编译检查或局部逻辑验证，应明确说明：

- 哪些部分已验证
- 哪些部分依赖真实设备
- 哪些副作用目前无法在本地确认
- 还需要什么环境或步骤完成最终验证

在这个项目里，诚实说明验证边界，比给出过度确定的结论更重要。

### 7.7 任务测试失败时，优先按链路分层定位

当任务测试失败时，建议不要直接把失败归因到“代码改坏了”。这个项目更常见的问题是链路中某一层没有准备好。建议按以下顺序定位：

1. 前端资源层
   页面资源是否成功加载，LuCI 模板变量是否正确注入。
2. LuCI 代理层
   `uhttpd` 和 `istore_backend.lua` 是否把请求正确代理到后端，登录态是否被正确透传。
3. 后端进程层
   quickstart 二进制是否存在、是否正在运行、是否监听预期端口。
4. 任务与部署层
   `.it-runner` 任务是否把正确的二进制部署到正确的设备路径，服务是否真正重启。
5. 系统副作用层
   请求即使成功返回，实际 `ubus/uci/shell` 动作是否真正生效。

如果能按这个顺序排查，很多“接口看起来坏了”的问题会更快收敛。

## 8. 重构时的约束与优先拆分思路

QuickStart 后续如果要做系统性重构，目标不应是“尽快把代码变新”，而应是先把边界、依赖和副作用理清，再逐步降低耦合。这个项目的复杂度主要不在语法层，而在前后端协作、设备环境依赖和流程编排逻辑。AI 在参与重构时，建议优先做结构化拆分，而不是一次性改写。

### 8.1 重构的首要约束是保持外部行为稳定

在没有单独批准的情况下，重构默认不应主动改变以下内容：

- 现有 API 路径
- 现有请求与响应结构
- LuCI 页面挂载方式
- 前端路由入口与资源加载方式
- 设备动作的触发语义
- `quickstart` 配置字段含义

原因很简单：这些内容往往已经被另一侧代码、发布流程、设备页面或用户习惯隐式依赖。
如果要改变，应作为独立任务明确提出，而不是夹带在“代码整理”中完成。

### 8.2 先拆边界，再拆实现

更稳妥的重构顺序通常是：

1. 先识别边界
   明确哪些代码属于：
   - 前端页面层
   - 前端请求层
   - 后端 API 层
   - 后端业务编排层
   - 设备适配层
2. 再识别副作用
   标出哪些逻辑最终会：
   - 修改 `uci`
   - 调用 `ubus`
   - 执行 shell 命令
   - 影响网络、磁盘、服务或首次流程状态
3. 最后才考虑内部抽象
   只有在边界和副作用清楚后，才适合抽公共函数、拆文件、改命名或统一接口风格。

不要先做“统一风格”，再去补理解。

### 8.3 优先拆“协调层”，谨慎拆“设备适配层”

这个项目里最适合优先重构的通常不是最底层命令调用，而是中间的“协调层”代码，也就是：

- 把多个设备动作组织成一个流程的 service
- 同时读写多个配置并返回一个综合结果的逻辑
- 同时承担参数解析、业务判断、命令拼接的函数

这些地方往往最难维护，也最容易形成巨大的隐式依赖。
相反，已经明显带有平台差异的设备适配层，不应因为“看起来重复”就仓促统一。

### 8.4 优先建立稳定映射关系

在真正大改之前，建议先补齐几类映射关系：

- 页面 / 组件 -> 请求接口
- 请求接口 -> 后端路由
- 后端路由 -> service 函数
- service 函数 -> `ubus/uci/shell`
- `backend/yamls` 中的 YAML / Swagger -> models / 前端类型

这些映射关系一旦清楚，后续不论是局部重构还是模块拆分，决策成本都会大幅下降。

### 8.5 将“可验证的纯逻辑”与“依赖真实设备的逻辑”分开

重构时应尽量把以下两类内容拆开：

- 纯参数处理、状态组装、规则判断
- 真实设备调用、系统配置读写、命令执行

前一类更容易单测、复用和安全重构。
后一类更适合通过窄接口隔离，减少误改面。
如果能把“业务判断”从“系统落地动作”中分离出来，项目可维护性会明显提升。

### 8.6 前后端重构不要同步大改，除非目标就是协议升级

同时重构前端和后端，风险很高。
更稳妥的方式通常是：

- 先稳定接口边界，再重构前端内部
- 或先整理后端内部，再保持接口不变
- 如果确实要改协议，明确作为一次“接口升级任务”单独处理

不要在一次普通重构里同时完成：

- 页面重组
- 接口字段改名
- 返回语义变化
- 设备动作时序调整

这样很难定位回归来源。

### 8.7 重构建议以渐进式阶段组织

后续如果正式推进重构，建议按阶段推进，而不是按“文件太乱”来切：

- 第一阶段：建立地图和边界
- 第二阶段：抽离纯逻辑与副作用逻辑
- 第三阶段：整理 API / models / 文档一致性
- 第四阶段：按模块域逐步拆分
- 第五阶段：必要时再考虑协议升级或前端体验重构

这样更符合这个项目的真实风险结构。

### 9.7 建议的补充使用方式

为了让 AI 输出更稳定，实际使用时建议再附加以下信息：

- 当前希望修改的是哪个功能域
- 涉及哪个目录或层，或是否怀疑前后端联动
- 是否允许改接口
- 是否有真实设备验证条件
- 当前是否通过 `.it-runner` 执行测试或部署
- 当前问题是否发生在浏览器访问 LuCI 路径时，而不是直接访问后端端口
- 是做小改动、排查问题，还是为重构做准备

这样 AI 更容易给出与项目真实风险匹配的方案。
